# 知识日报 — 2026 年 5 月 15 日

## MCP 协议：AI Agent 工具调用的新标准

Anthropic 近期开源的 **Model Context Protocol (MCP)** 正在成为 AI Agent 工具集成的事实标准。MCP 定义了一套统一的接口规范，让 LLM 应用能够以标准化的方式接入外部工具和数据源。

### 核心设计

MCP 采用客户端-服务器架构：

```
┌─────────────┐     ┌──────────────┐     ┌──────────────┐
│  LLM 应用   │────▶│  MCP 客户端  │────▶│  MCP 服务器  │
│  (Host)     │     │  (Client)    │     │  (Server)    │
└─────────────┘     └──────────────┘     │  - 文件系统  │
                                          │  - 数据库    │
                                          │  - API       │
                                          └──────────────┘
```

关键特性：
- **标准化工具定义**：所有工具通过 JSON Schema 描述输入输出
- **资源（Resources）**：服务器可暴露文件、数据库记录等资源，支持类似文件系统的 URI 寻址
- **提示模板（Prompts）**：预定义的提示词模板，方便复用

### 与 Function Calling 的对比

| 维度 | MCP | 传统 Function Calling |
|------|-----|----------------------|
| 协议层级 | 应用层协议 | API 层面的约定 |
| 工具发现 | 动态（list_tools） | 静态预定义 |
| 资源模型 | 统一（Resources） | 无标准模型 |
| 可组合性 | 多服务器可组合 | 单一定义 |
| 生态 | 开源，社区驱动 | 各厂商独立实现 |

MCP 的核心价值在于将工具调用从"硬编码"变为"可发现"——Agent 可以动态查询可用工具，而不是在系统提示中静态声明。

### 实践案例

以文件操作为例，MCP 服务器的定义模式：

```python
# MCP 服务器示例：文件搜索工具
@mcp.tool()
async def search_files(pattern: str, path: str = ".") -> list[str]:
    """递归搜索匹配模式的文件"""
    return glob.glob(f"{path}/**/{pattern}", recursive=True)
```

## RAG 评估框架：RAGAS 2.0

RAGAS（RAG Assessment）近期发布 2.0 版本，新增多项评估指标：

- **上下文精度（Context Precision）**：衡量检索结果中相关文档的排序质量
- **答案忠实度（Answer Faithfulness）**：检查生成答案是否与检索上下文一致
- **引用召回（Citation Recall）**：验证答案中的每个声称是否有对应引用

RAGAS 2.0 的差异化之处在于引入了 **基于 LLM 的自动评判**（LLM-as-Judge），用 GPT-4 或 Claude 等模型对 RAG 管道的各环节进行打分。

```python
from ragas import evaluate
from ragas.metrics import context_precision, faithfulness

result = evaluate(
    dataset=test_dataset,
    metrics=[context_precision, faithfulness],
    llm=evaluator_model
)
print(result)  # 输出各维度的得分
```
