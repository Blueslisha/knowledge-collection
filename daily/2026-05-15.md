## 一、MCP 协议：AI Agent 工具调用的标准化范式

### 1.1 背景与动机

Anthropic 于 2024 年底开源的 **Model Context Protocol (MCP)**，正在从一项实验性规范演变为 AI Agent 工具集成领域的事实标准。MCP 出现的根本原因在于：LLM 应用接入外部工具的方式长期处于"各自为政"的状态——OpenAI 有 Function Calling，Google 有 Tool Use，Anthropic 有 Tool Use API，每个厂商定义工具的方式、传输协议、错误处理机制各不相同。这种碎片化导致工具提供方需要为每个平台单独适配，严重制约了 Agent 生态的发展。

MCP 的核心思路是借鉴 **LSP（Language Server Protocol）** 的成功经验——正如 LSP 统一了编辑器与语言服务器的通信协议，MCP 试图统一 LLM 应用与工具/数据源之间的通信协议。

### 1.2 协议架构

MCP 采用典型的客户端-服务器架构，包含三个核心角色：

```
┌─────────────────┐      JSON-RPC       ┌──────────────────┐
│                 │ ◄──────────────────► │                  │
│   LLM 应用      │      (stdio/SSE)     │   MCP 服务器      │
│   (Host)        │                      │   (Server)        │
│                 │                      │                  │
│  ┌───────────┐  │                      │  ┌────────────┐  │
│  │ MCP 客户端 │  │                      │  │ 工具定义    │  │
│  │ (Client)   │  │                      │  │ 资源暴露    │  │
│  └───────────┘  │                      │  │ 提示模板    │  │
└─────────────────┘                      │  │ 传输层      │  │
        │                                │  └────────────┘  │
        │                                └──────────────────┘
        ▼
  ┌─────────────┐
  │ 用户界面     │
  └─────────────┘
```

**Host（主机）**：运行 LLM 应用的进程，负责处理用户输入、协调模型调用。典型例子包括 Claude Desktop、VS Code 插件、自定义 Agent 框架。

**Client（客户端）**：在 Host 进程中运行的协议客户端，与 MCP Server 建立一对一的连接。每个 Client 对应一个 Server 实例。

**Server（服务器）**：提供具体工具、资源和提示模板的后端进程。Server 通过标准化的接口暴露能力。

### 1.3 传输层

MCP 目前支持两种传输方式：

| 传输方式 | 适用场景 | 优缺点 |
|---------|---------|--------|
| **stdio** | 本地子进程通信 | 延迟低、无需网络；不适合分布式部署 |
| **SSE (Server-Sent Events)** | 远程服务器 | 支持 HTTP，兼容性好；单向推送需配合请求通道 |

实际的标准化传输协议定义如下：

```
// 客户端 → 服务器 (请求)
Content-Type: application/json

{"jsonrpc": "2.0", "id": 1, "method": "tools/call", "params": {
  "name": "search_files",
  "arguments": {"pattern": "*.py", "path": "/src"}
}}

// 服务器 → 客户端 (响应)
{"jsonrpc": "2.0", "id": 1, "result": {
  "content": [
    {"type": "text", "text": "/src/main.py\n/src/utils.py"}
  ]
}}
```

### 1.4 核心协议能力

MCP 定义了三个核心能力领域，每个领域都有对应的生命周期方法：

#### 工具（Tools）

工具是 Server 暴露的可调用函数，每个工具通过 JSON Schema 描述其输入参数。

标准方法：
- `tools/list` — 返回 Server 支持的所有工具列表
- `tools/call` — 调用指定工具并传入参数

工具响应支持多种内容类型：`text`、`image`、`resource`（资源引用），使工具不仅能返回文本，还能返回图片或引用其他资源。

#### 资源（Resources）

资源是 Server 暴露的数据实体，采用类似文件系统的 URI 方案进行寻址。

```
file:///home/user/config.json
postgres://database/orders?schema=public
git://repo/main/README.md
```

标准方法：
- `resources/list` — 列出可用资源
- `resources/read` — 读取指定资源的内容
- `resources/subscribe` — 订阅资源变更通知

资源系统的设计使 AI Agent 能够像人类使用文件系统一样发现和读取数据，极大降低了 Agent 与数据源集成的复杂度。

#### 提示模板（Prompts）

提示模板是预定义的提示词模板，Server 可以暴露专业领域的提示模板供 Host 使用。

```
prompts/list → 返回模板列表
prompts/get → 获取具体模板内容（含参数替换）
```

例如，一个数据库 MCP Server 可以提供："编写 SQL 查询"、"分析表结构"、"优化慢查询"等专用提示模板。

### 1.5 MCP vs 传统 Function Calling

从协议层面审视 MCP 与传统 Function Calling 的差异：

| 维度 | MCP | OpenAI Function Calling |
|------|-----|----------------------|
| **工具发现** | 运行时动态查询 (`tools/list`) | 编译时静态声明（系统 prompt 中定义） |
| **工具更新** | 随时添加/移除，无需重启 LLM 应用 | 需要更新 prompt 或重新初始化 |
| **通信协议** | 标准化 JSON-RPC 2.0 | HTTP API 调用，无统一协议 |
| **资源模型** | 统一 URI 方案，支持资源发现 | 无，需要额外实现 |
| **状态管理** | 支持 Subscribe/Notification 模式 | 无内置，需自建轮询 |
| **组合性** | 多 Server 可组合，按需加载 | 单一定义，组合需自行编排 |
| **安全边界** | 每个 Server 独立进程，自然隔离 | 取决于调用实现 |
| **生态标准** | 开源，社区驱动，LSP 模式 | 厂商锁定 |

### 1.6 工程实践：构建自定义 MCP Server

```python
# server.py — 一个简单的文件搜索 MCP Server
from mcp.server import Server, NotificationOptions
from mcp.server.models import InitializationOptions
import mcp.server.stdio
import mcp.types as types
import glob
import os

# 创建 Server 实例
server = Server("file-search")

@server.list_tools()
async def handle_list_tools() -> list[types.Tool]:
    return [
        types.Tool(
            name="search_files",
            description="递归搜索匹配模式的文件",
            inputSchema={
                "type": "object",
                "properties": {
                    "pattern": {
                        "type": "string",
                        "description": "文件匹配模式（如 *.py, test_*）"
                    },
                    "base_path": {
                        "type": "string",
                        "description": "搜索根路径，默认为当前目录",
                    }
                },
                "required": ["pattern"]
            }
        ),
        types.Tool(
            name="read_file",
            description="读取指定文件的内容",
            inputSchema={
                "type": "object",
                "properties": {
                    "path": {"type": "string"}
                },
                "required": ["path"]
            }
        )
    ]

@server.call_tool()
async def handle_call_tool(
    name: str, arguments: dict
) -> list[types.TextContent]:
    if name == "search_files":
        pattern = arguments["pattern"]
        base_path = arguments.get("base_path", ".")
        matches = glob.glob(
            f"{base_path}/**/{pattern}",
            recursive=True
        )
        return [types.TextContent(
            type="text",
            text="\n".join(matches) if matches else "无匹配结果"
        )]
    elif name == "read_file":
        path = arguments["path"]
        if not os.path.isfile(path):
            return [types.TextContent(
                type="text", text=f"文件不存在: {path}"
            )]
        with open(path, "r", encoding="utf-8") as f:
            content = f.read()
        return [types.TextContent(type="text", text=content)]
    raise ValueError(f"未知工具: {name}")

async def main():
    async with mcp.server.stdio.stdio_server() as (read, write):
        await server.run(read, write, InitializationOptions(
            server_name="file-search",
            server_version="0.1.0",
        ))

if __name__ == "__main__":
    import asyncio
    asyncio.run(main())
```

## 二、RAGAS 2.0：RAG 评估框架深度解析

### 2.1 为什么需要专用评估框架

RAG 系统的评估一直是个棘手问题。与传统检索系统不同，RAG 管道包含两个耦合的环节——检索（Retrieval）和生成（Generation）——任一环节的缺陷都会影响最终输出质量。更复杂的是，这两个环节可能相互掩盖问题：生成模型可能"脑补"出检索结果中没有的信息（幻觉），也可能因为检索结果太差而无法生成有用答案。

RAGAS（RAG Assessment）2.0 提供了一套系统化的评估方法论，从多个独立维度对 RAG 管道进行量化评估。

### 2.2 评估维度全景

RAGAS 2.0 的评估指标可分为三大类：

#### 检索质量

| 指标 | 定义 | 计算方式 |
|------|------|---------|
| **Context Precision** | 检索结果中相关文档的排序质量 | 计算相关文档在排序中的位置加权得分 |
| **Context Recall** | 需要的上下文信息有多少被成功召回 | 基于 ground truth 计算召回率 |
| **Context Relevancy** | 检索结果的噪声比例 | 评估检索文档中与问题无关的噪声占比 |

#### 生成质量

| 指标 | 定义 | 计算方式 |
|------|------|---------|
| **Faithfulness** | 生成答案是否忠实于检索上下文 | 将答案拆解为原子声称，逐一验证是否可被上下文支持 |
| **Answer Relevancy** | 生成的答案与问题的相关性 | 评估答案是否直接回应了问题的核心诉求 |
| **Answer Correctness** | 答案的事实正确性 | 与 ground truth 对比的精确匹配或语义相似度 |
| **Citation Recall** | 每个声称是否有对应引用 | 检查答案中的事实性声称是否附带了来源引用 |

#### 端到端质量

| 指标 | 说明 |
|------|------|
| **Harmfulness** | 输出是否包含有害内容 |
| **Coherence** | 答案的连贯性和逻辑一致性 |
| **Fluency** | 语言流畅度 |

## 三、本周值得关注的学术论文

### 3.1 LongRoPE: 将 LLM 上下文扩展至 2048K

**核心问题**：RoPE 位置编码在超出训练长度时会出现 perplexity 急剧上升的问题。现有扩展方法（如线性插值、NTK-aware 插值）在扩展到 128K-256K 时质量尚可，但进一步扩展时失效。

**LongRoPE 的解决方案**：
1. **非线性插值**：对不同频率分量采用不同的插值策略——低频分量（负责长距离依赖）大幅压缩，高频分量（负责局部位置）保持精度
2. **渐进式训练**：先扩展到 256K，再基于 256K 的 checkpoint 扩展到 2048K，避免一次到位带来的优化困难
3. **温度缩放**：在注意力 softmax 中引入温度参数，缓解长上下文中的注意力分布过度平滑

**实验结果**：
| 上下文长度 | 标准 RoPE | Linear | NTK-aware | LongRoPE |
|-----------|-----------|--------|-----------|----------|
| 128K | 45.2 | 18.3 | 15.7 | **12.1** |
| 512K | 128+ | 42.6 | 38.4 | **14.8** |
| 2048K | — | — | — | **18.6** |

Perplexity 越低越好。LongRoPE 在 2048K 上下文下的 perplexity 仅比基线 4K 上下文高出约 3%，基本实现了无损扩展。

### 3.2 CodeAgent: 自主代码生成 Agent

**来源**：Microsoft Research

**架构创新**：
- 采用分层规划策略（Hierarchical Planning）：顶层 Agent 将复杂任务分解为子任务，底层 Agent 逐个执行
- 安全执行沙箱：代码在隔离容器中运行，通过文件系统和网络调用追踪生成完整的执行轨迹
- 自我纠错循环：当执行出错时，Agent 自动分析错误日志并修改代码，最多重试 3 次

**SWE-Bench 结果**：

| 方法 | 通过率 | 平均执行时间 |
|------|-------|-------------|
| GPT-4 (zero-shot) | 12.7% | — |
| SWE-Agent | 34.8% | 124s |
| Devin | 41.2% | 215s |
| **CodeAgent** | **43.1%** | **96s** |

CodeAgent 用更少的执行时间取得了更高的通过率，核心原因在于其分层规划减少了不必要的代码生成尝试。

### 3.3 Hard Negative Mining 在稠密检索中的应用

**来源**：SIGIR 2026

稠密检索面临的关键挑战是：**负样本的选择质量直接影响模型效果**。简单随机采样的负样本与 query 差异过大，模型无法学到精细的区分能力。

该论文提出的方法：
1. **多阶段难负样本挖掘**：第一阶段用 BM25 召回候选负样本，第二阶段用当前模型筛选与 query 语义相似但不相关的样本作为负样本
2. **动态更新**：每 N 步训练后重新生成难负样本池，避免模型过拟合到固定的负样本集
3. **边际损失函数**：

```
L = max(0, sim(q, d⁺) - sim(q, d⁻) + margin)
```

其中 margin 根据负样本难度动态调整——难度越高的负样本，margin 越小。

**效果**：
- Recall@10 提升 7.2%
- NDCG@10 提升 5.8%
- 训练收敛速度加快 2.3 倍
